@troll-kadabra I was going to answer you on discourse but you've deleted the thread, I shouldn't answer you here not only because it doesn't help other people with the same problem but also because of the internet points that I'll not get :laughing:.
Just expose what you want manually. I've most of GameWorks working flawlessly in AtomicNET (what's awesome, yay! :smile:). Build master (all) and start from there, to expose a field, use the files in the "Generated" folder of AtomicNETNative as a template, say for example you want to expose an int that's member of a scene, in Scene.h add:

    int x_;
    void sxm(int x) { x_ = x; }
    int gxm() { return x_; }

Then, in CSModuleScene.cpp, add:

ATOMIC_EXPORT_API void csb_Atomic_Scene_sxm(Scene* self, int x) { self->sxm(x); }
ATOMIC_EXPORT_API int csb_Atomic_Scene_gxm(Scene* self) { return self->gxm(); }

Build AtomicNETNative.
To CSModuleScene.cs in AtomicNET.shproj, add thw following:

[DllImport(Constants.LIBNAME, CallingConvention = CallingConvention.Cdecl, CharSet = CharSet.Ansi)]
private static extern void csb_Atomic_Scene_sxm(IntPtr self, int x);
public void sxm (int x) { csb_Atomic_Scene_sxm(nativeInstance, x); }

[DllImport(Constants.LIBNAME, CallingConvention = CallingConvention.Cdecl, CharSet = CharSet.Ansi)]
private static extern int csb_Atomic_Scene_gxm(IntPtr self);
public int gxm() { return csb_Atomic_Scene_gxm(nativeInstance); }

Build it and reference the dll in your AtomicNET project. An easy way to get it working is creating a new C# project in the editor and modifying it, point AtomicNETNative to the one you just build (don't forget to set it to copy if new on properties). Remove the AtomicNET projects that are setup by default, and if you wish you can add the AtomicNET project generated in the "AtomicGameEngine\Artifacts\AtomicNET" and add as dependency so it builds before your project and also it's easier for editing. You can also add the C++ projects (Atomic and AtomicNETNative) there, luckily Visual Studio allows you to have C++ and C# projects in the same solution what's really nice.
For default mashallings check the Pinvoke conversion table. You can pass complex types but that's out of the scope of this tutorial and you said you only need primitives anyway.
Don't count on the bindings generators to do crazy stuff for you, @JoshEngebretson did an awesome job but they can't read your mind. Make your changes in a branch so you can easily rebase.
For a more complete bindings generator check SWIG, I've been using it with great success, the modern versions (>=3) are awesome, you don't even need an interface file, you just point it to your headers and it works almost perfectly.
Hope this helps somehow. My precious internet points :cry:.
